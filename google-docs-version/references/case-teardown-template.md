# 案例拆解文档模板（Google Docs / 本地 DOCX 版）

按下面标题顺序写，不要随意调整。标题维度固定，但每个案例的具体表现不固定。不要把某一个案例的路径、渠道、转化方式、奖励方式硬套到其他案例里。

案例命名、文档标题、小节里引用的案例名称都用中文。品牌或产品官方英文名可以保留，但必须配中文机制描述，例如“拼多多现金大转盘案例拆解”，不要只写 `PDD cash wheel` 或其他英文 slug。

写作默认带入高级用户增长产品经理视角：把案例当成竞品来反推产品设计，而不是只描述现象。拆解时要看玩法、路径、文案、交互、奖励数值、门槛、风控、数据口径和复用方案。

## 概述

作用：先给结论，让读者一眼看懂这个案例最值得研究的地方，以及它对我们有什么用。

写法：
- 用一句加粗的话概括这个案例的核心机制。
- 列出 2-4 个最值得复用的产品能力、运营设计或增长机制。
- 先写结论，再补背景。
- 不要默认它一定是裂变、付费、私域、SaaS 或社群案例，要根据实际证据判断。

## 研究边界与方法

作用：说明这篇拆解研究了什么、没研究什么、证据怎么分层，避免把事实和猜测混在一起。

建议用 Google Docs 或 Word 真实表格写：
- 研究对象：品牌、产品、活动、页面、App、社群、广告、视频、直播间、门店或其他真实对象。
- 研究范围：本次实际看到了哪些页面、流程、截图、视频片段、公开资料或后台线索。
- 未覆盖范围：哪些内容没有证据，比如真实投放数据、后台规则、财务数据、完整用户规模等。
- 研究维度：用户路径、规则设计、玩法解析、UI 交互分析、增长内核与归因、SaaS 后台逻辑、如何复用、商业闭环、不足分析。
- 信息分层：没有直接证据支撑的内容统一标 `[推断]`。
- 数据原则：所有数字都写清来源、日期、口径；没有来源就不要写成事实。

## 用户完整路径时序图

作用：先用一张真实截图路径图展示用户从进入到完成关键动作的完整过程，再进入文字拆解。

这个部分必须保留。最终图必须是“真实案例截图 + 每一步用户操作备注”的横向故事板，不要只输出抽象流程图或系统时序图。

Google Docs 或本地 DOCX 正文里必须放真实截图故事板图片，位置就在这个标题下方。Google Docs 图片插入必须通过 connector 支持的图片写入路径，或通过 DOCX-first 导入路线完成；本地 DOCX 必须把图片作为 Word 图片对象插入。不能把本地图片路径、Markdown 图片语法或文字占位符当成已经插图。

在线文档会把大图缩放到页面宽度，所以长路径不要只放一张十几步总览图。标准做法是：

- 每张分段故事板最多 2 步。
- 单个手机截图源宽度保留约 720px。
- 遇到金额明细、规则说明、列表记录等小字关键页，正文里补 1 步 1 张的原始单屏截图。
- 完整总览图只用于看顺序，不能作为唯一流程图。
- 正文优先插入 `01-02`、`03-04` 这类分段高清图。

路径图按真实案例来定，不要硬套固定环节。通常可以覆盖：
- 用户第一次从哪里看到这个案例或入口。
- 用户进入了哪个页面、内容、活动、商品、服务或工具。
- 用户做了什么关键动作，比如浏览、点击、领取、注册、授权、加购、下单、留资、分享、进群、下载、关注、预约等。
- 系统、页面、消息或人工服务给了什么反馈。
- 用户如何被引导到下一步。
- 用户完成的关键目标动作是什么。
- 后续闭环是什么，可能是复访、留存、复购、传播、线索沉淀、付费、进群、咨询、下载或其他结果。

## 1. 用户路径

作用：按顺序说明用户看到了什么、做了什么、为什么会继续走下去。

每个小节都必须先写文案说明，再贴对应步骤截图。截图要紧跟小节文字，不要集中放到文末，也不要只依赖“用户完整路径时序图”的总览图。

推荐结构：

```markdown
### 1.x 阶段名称

文案说明：用户看到了什么、做了什么、页面如何引导、底层逻辑是什么。

[这里插入对应步骤截图；如果分析动态交互，这里也要插入对应 GIF 或连续关键帧]

图注：这张图证明了哪个入口、文案、交互或进度设计。
```

每一步都要写：
- 用户动作。
- 页面、消息、视频帧或截图证据。
- 关键文案或关键 UI 机制。
- 关键动态交互，比如弹窗、提示条、动画、按钮状态变化、进度变化。
- 这一步的业务目的。
- 证据不足时标 `[推断]`。

## 2. 规则设计

作用：把案例背后的规则拆出来，说明用户为什么会按这个路径行动。

规则不一定都是奖励或邀请，也可能是价格、门槛、时间、身份、库存、任务、内容分发、服务承诺、社交关系或平台限制。根据案例实际情况看：
- 进入规则：谁能参与，入口在哪里，有没有门槛。
- 行动规则：用户需要完成什么动作，动作顺序是否被设计过。
- 激励规则：有没有奖励、折扣、权益、内容、身份、排名、稀缺名额等。
- 约束规则：有没有时间限制、次数限制、资格限制、库存限制、地域限制等。
- 传播规则：有没有分享、邀请、拼团、助力、转发、口碑扩散等设计。
- 留存规则：有没有签到、任务、订阅、复访、提醒、连续参与等设计。
- 防滥用规则：有没有能看到的防作弊、风控、合规或审核设计。

要分清楚哪些是直接观察到的，哪些只是 `[推断]`。

## 3. 玩法解析

作用：解释用户为什么愿意参与、为什么愿意继续、为什么愿意完成目标动作。

每个小节都要先讲机制，再补一段“底层逻辑分析”。如果这一步有动态 UI，还要补一段“UI 交互分析”。最后贴对应截图、GIF 或连续关键帧。证据必须放在对应小节文字下方，不能只放在总览图、附件或文末。

重点看真实案例里出现的动机，不要预设一定是某一种玩法。可关注：
- 入口诱因：用户为什么会点进去。
- 利益驱动：用户获得什么好处。
- 信任机制：用户为什么相信它。
- 降低阻力：流程是否短、说明是否清楚。
- 紧迫感：是否有倒计时、限量、限时、库存、名额、阶段任务等。
- 社交机制：是否利用熟人关系、群体氛围、从众、攀比、协作或竞争。
- 内容机制：是否通过视频、图文、直播、教程、测评、评论、榜单等内容推动用户。
- 渠道闭环：用户是否被某个渠道持续承接。

底层逻辑不能硬套名词。必须说明它对应到截图里的哪个文案、数字、按钮、进度、奖励或页面反馈；证据不足时标 `[推断]`。

UI 交互分析要拆清楚这几件事：
- 触发条件：用户做了什么，或系统在哪个状态下弹出。
- 出现位置：它盖住了哪里，靠近哪个关键金额、按钮、任务或进度。
- 视觉强调：颜色、动效、遮罩、红框、震动、音效、数字变化、按钮状态如何吸引注意。
- 用户下一步：它希望用户点什么、继续什么、放弃什么。
- 业务目的：它是在催促、解释、降低疑虑、制造紧迫感、强化收益，还是做风控提示。
- 打扰和风险：是否遮挡主路径、是否误导、是否让用户反感，证据不足时标 `[推断]`。

例如“今日提现峰值提醒”这种顶部提示，不能只写“系统提醒用户”。要写清楚它为什么在用户接近 50 元提现门槛时出现，如何借“当前处于提现高峰期、已有大量用户提现成功”制造紧迫感和社交证明，并把用户注意力拉回“今日成功提现”。

只要写了 UI 交互分析，就必须在该小节下方贴对应 GIF 或连续关键帧截图。不能只写“见用户完整路径时序图”，也不能只把 GIF 放在本地文件或附件里。

## 4. 增长内核与归因

作用：把真正带来增长的原因和表面动作分开。

重点写：
- 核心增长引擎：到底是什么在驱动新增、留存、转化或传播。
- 增长循环：用户行为如何带来下一批用户、下一次访问、下一次购买或下一次内容消费。
- 关键杠杆：价格、渠道、内容、关系链、产品体验、供给、服务、品牌、平台流量等，哪个最关键。
- 归因拆分：哪些增长来自渠道，哪些来自产品机制，哪些来自运营动作，哪些来自外部环境。
- 证据等级：有数据的写事实，没有数据的标 `[推断]`。

## 5. SaaS 后台逻辑

作用：把案例翻译成可复用的后台能力，方便后续做产品设计或系统设计。

根据案例实际情况拆：
- 用户与身份。
- 内容与页面。
- 规则配置。
- 关系与传播。
- 消息触达。
- 交易与转化。
- 数据与归因。
- 风控与合规。

只有在目标技术栈明确时，才补技术映射。不要为了显得完整强行写技术栈。

## 6. 如何复用

作用：把案例变成我们可以执行的产品方案、运营动作或系统能力。

如果用户指定目标行业或业务，例如 casino、电商、游戏、SaaS、金融、教育等，本节必须按该行业重写，不能只写通用复用点。

重点写：
- 哪些机制可以直接复用。
- 哪些机制只能改造后复用。
- 哪些机制不能复用，原因是什么。
- 复用到我们的产品里应该做成什么功能、页面、规则、内容、后台配置或运营流程。
- 不同市场、不同渠道、不同用户层级下需要怎么改。
- 第一版应该做多小，先验证什么。

## 7. 商业闭环

作用：说明这个案例如何从用户行为变成业务结果。

根据案例实际情况写：
- 用户完成的业务目标是什么。
- 这个目标如何变成收入、成本下降、线索沉淀、用户资产或长期价值。
- 有没有免费入口、低价入口、付费产品、套餐、佣金、广告、服务费、订阅、复购等结构。
- 已验证的数据是什么；没有数据就标 `[推断]`。
- 用户路径、留存、传播和商业结果之间如何互相影响。

## 8. 不足分析

作用：在复用前先看清风险，避免只看到成功的一面。

重点看：
- 证据不足。
- 平台依赖。
- 人或品牌依赖。
- 供给依赖。
- 市场限制。
- 合规风险。
- 产品限制。
- 数据限制。
- 护城河强弱。

如果案例涉及奖励、返利、抽奖、助力、邀请、游戏化、虚拟币、下注、充值或提现，要补一段完整风控措施，至少覆盖身份与设备、行为异常、奖励安全、合规边界、运营处置。

## 对我们 SaaS 产品的核心启发

作用：把拆解结论变成产品优先级。

建议按优先级写：
- P0：最应该先做，能最快验证核心价值。
- P1：第二层能力，能提高效率、转化或复用范围。
- P2：后续可选能力，等 P0/P1 验证后再做。

每一项都要写：
- 功能名称。
- 对应案例里的哪个证据或机制。
- 为什么重要。
- 建议归属模块或负责人。
- 第一版最小可用功能应该包含什么。
- 验证指标是什么。

## Google Docs / 本地 DOCX 交付注意事项

输出到 Google Docs 或本地 DOCX 时，要保护目标文档结构：

- 不要默认整篇覆盖。
- 不要为了插图方便删除后半篇再重建。
- 不要把本地图片路径或 Markdown 图片语法写进正文后声称已插图。
- 新建文档时，可以先写正文结构，再按目标文档支持能力插入故事板和小节截图。
- 需要表格时，用 Google Docs 或 Word 真实表格；Google Docs 要用 connector 读回确认行列和单元格内容，本地 DOCX 要重新读取文件确认表格数量和内容。
- 需要图片时，Google Docs 必须通过 connector 可验证的图片对象或 DOCX-first 导入完成；本地 DOCX 必须把图片作为 Word 图片对象插入。
- 如果当前工具无法插入或读回图片，要停下来说明限制，不能用文字占位符冒充图片。
- 用户要求本地 DOCX 时，最终交付本地 `.docx` 文件路径，不要写 Google Docs 链接占位，也不要说已经写入在线文档。
