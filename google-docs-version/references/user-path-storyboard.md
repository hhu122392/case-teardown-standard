# 用户路径故事板规范（Google Docs / 本地 DOCX 版）

## 基本形态

最终交付必须优先做成“真实截图 + 备注”的横向路径故事板，样式参考：

`../assets/user-path-storyboard-standard.png`

这张图是“用户完整路径时序图：真实截图 + 每一步操作备注”的输出标准。后续做案例拆解时，路径图至少要达到这个信息密度：真实截图、步骤编号、用户动作标题、业务备注、箭头顺序都要齐。

这不是抽象的系统时序图。Mermaid 只能用来先梳理顺序，不能替代最终故事板。

## 每一步要写什么

每个步骤都要写清楚：

- 真实截图。
- 用户动作，例如“用户点击分享按钮”。
- 系统反应或页面变化，例如“弹出分享面板”。
- 业务目的，例如“降低分享操作成本”。

备注优先写用户动作，不要只写页面名称。

## 文档清晰度标准

Google Docs、飞书云文档、在线 Word、本地 Word 这类文档都会把插入图片缩放到页面宽度。如果 12-20 个步骤只做成一张总图，每个手机截图会被压得很小；导入后即使源图没坏，读者也会觉得糊。

所以 Google Docs / 本地 DOCX 版必须按下面标准产出：

- 单个手机截图源宽度保留到 `720px` 左右，不要压到 200-300px。
- 每张故事板最多放 2 个步骤。长路径要输出多张分段图，例如 `01-02`、`03-04`、`05-06`。
- 如果截图里有金额明细、规则说明、列表小字这类必须读清的内容，优先改成 1 步 1 张图，或者在正文小节下方补一张原始单屏截图。
- 可以另外输出一张完整总览图，但它只用于看路径顺序，不能作为文档里唯一的用户流程图。
- 插入文档时，优先插入分段高清图；总览图放在前面或附录都可以。
- 故事板图片用 PNG，不要转成低质量 JPEG。
- 正文小节截图仍然要保留，不能用故事板总览替代。

如果文档最终只允许放一张图，宁可减少步骤数量，也不要把十几步压成一张小字总图。

## 本地故事板图片

已有截图和备注时，用本版本脚本生成本地横向图片：

```powershell
$skillRoot = "<google-docs-version skill root>"
python (Join-Path $skillRoot "scripts\build_user_path_storyboard.py") `
  --items ".\storyboard-items.tsv" `
  --output ".\user-path-storyboard.png"
```

这个命令会同时输出：

- `user-path-storyboard.png`：完整总览图，用于快速看全路径。
- `user-path-storyboard-01-02.png`、`user-path-storyboard-03-04.png` 等：分段高清图，用于插入 Google Docs 或本地 DOCX 正文。

如果小字仍然看不清，改用下面这个命令输出 1 步 1 张图：

```powershell
$skillRoot = "<google-docs-version skill root>"
python (Join-Path $skillRoot "scripts\build_user_path_storyboard.py") `
  --items ".\storyboard-items.tsv" `
  --output ".\user-path-storyboard.png" `
  --columns 1 `
  --chunk-size 1
```

`storyboard-items.tsv` 每行 3 列，用 Tab 分隔：

```text
.\step1.png	用户点击页面入口	看到核心利益点，准备进入下一步
.\step2.png	用户完成关键动作	系统给出结果反馈或下一步引导
```

## Google Docs / 本地 DOCX 写入

推荐路径：

1. 先写入 `## 用户完整路径时序图` 标题和一句说明。
2. 把分段高清故事板图片作为 Google Docs 或 Word 图片对象插入这个标题下方；长路径不要只插入完整总览图。
3. 如果 connector 不支持直接插入本地图片，就按 `google-drive:google-docs` 的 DOCX-first 路线生成本地 DOCX，再导入成 native Google Doc。
4. 如果目标就是本地 DOCX，直接生成并验证 `.docx`，不需要再导入 Google Docs。
5. 如果图片无法插入或无法读回验证，只能在最终回复里说明限制，不能说故事板已经写入目标文档。

不能把这些内容当成完成：

- `![故事板](C:\path\user-path-storyboard.png)`
- `图片见本地文件`
- `TODO: 插入图片`
- 只有 Mermaid 流程图，没有真实截图故事板。
- 把十几步流程压成一张低清总图后当作唯一交付图。

## 正文小节截图不是总览图

故事板只解决“总览”。正式案例文档还必须在 `1. 用户路径` 和 `3. 玩法解析` 的每个小节下方放对应截图：

- 小节文字先解释发生了什么和为什么有效。
- 截图紧跟该小节文字。
- 图注说明这张图证明了什么。
- 不要把这些小节截图集中放到文末。

## 验证

写入后必须重新读回目标文档：

- Google Docs：重新读回 Google Doc，确认 `用户完整路径时序图` 标题存在。
- 本地 DOCX：重新读取 `.docx`，确认 `用户完整路径时序图` 标题存在。
- 确认分段高清故事板图片对象在这个标题附近。
- 确认 `1. 用户路径` 和 `3. 玩法解析` 的小节截图没有只留下本地路径或占位符。
- 确认故事板源图里单个手机截图没有被压到 300px 以下。
- 在线文档正文宽度内预览时，故事板里每个手机屏不能小于页面宽度的二分之一；如果小于这个比例，说明一张图里步骤还是太多。
- 如果只能验证文字，不能验证图片对象，要如实说明。
