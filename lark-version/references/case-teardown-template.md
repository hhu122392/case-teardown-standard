# 案例拆解文档模板

按下面标题顺序写，不要随意调整。标题维度固定，但每个案例的具体表现不固定。不要把某一个案例的路径、渠道、转化方式、奖励方式硬套到其他案例里。

案例命名、文档标题、小节里引用的案例名称都用中文。品牌或产品官方英文名可以保留，但必须配中文机制描述，例如“拼多多现金大转盘案例拆解”，不要只写 `PDD cash wheel` 或其他英文 slug。

写作默认带入高级用户增长产品经理视角：把案例当成竞品来反推产品设计，而不是只描述现象。拆解时要看玩法、路径、文案、交互、奖励数值、门槛、风控、数据口径和复用方案。

## 概述

作用：先给结论，让读者一眼看懂这个案例最值得研究的地方，以及它对我们有什么用。

写法：
- 用一句加粗的话概括这个案例的核心机制。
- 列出 2-4 个最值得复用的产品能力、运营设计或增长机制。
- 先写结论，再补背景。
- 不要默认它一定是裂变、付费、私域、SaaS 或社群案例，要根据实际证据判断。

## 研究边界与方法

作用：说明这篇拆解研究了什么、没研究什么、证据怎么分层，避免把事实和猜测混在一起。

建议用表格写：
- 研究对象：品牌、产品、活动、页面、App、社群、广告、视频、直播间、门店或其他真实对象。
- 研究范围：本次实际看到了哪些页面、流程、截图、视频片段、公开资料或后台线索。
- 未覆盖范围：哪些内容没有证据，比如真实投放数据、后台规则、财务数据、完整用户规模等。
- 研究维度：用户路径、规则设计、玩法解析、UI 交互分析、增长内核与归因、SaaS 后台逻辑、如何复用、商业闭环、不足分析。
- 信息分层：没有直接证据支撑的内容统一标 `[推断]`。
- 数据原则：所有数字都写清来源、日期、口径；没有来源就不要写成事实。

## 用户完整路径时序图

作用：先用一张真实截图路径图展示用户从进入到完成关键动作的完整过程，再进入文字拆解。

这个部分必须保留，并且优先用飞书画板输出。最终图必须是“真实案例截图 + 每一步用户操作备注”的横向故事板，不要只输出抽象流程图或系统时序图。

文档正文里也必须放同一张截图故事板图片，位置就在这个标题下方。画板可以作为可编辑版本，但不能替代正文里的截图故事板，因为读者阅读文档时需要直接看到真实路径。

路径图按真实案例来定，不要硬套固定环节。通常可以覆盖：
- 用户第一次从哪里看到这个案例或入口。
- 用户进入了哪个页面、内容、活动、商品、服务或工具。
- 用户做了什么关键动作，比如浏览、点击、领取、注册、授权、加购、下单、留资、分享、进群、下载、关注、预约等。
- 系统、页面、消息或人工服务给了什么反馈。
- 用户如何被引导到下一步。
- 用户完成的关键目标动作是什么。
- 后续闭环是什么，可能是复访、留存、复购、传播、线索沉淀、付费、进群、咨询、下载或其他结果。

每一步至少写清楚：
- 对应哪张真实截图。
- 用户看到了什么。
- 用户做了什么动作。
- 页面、系统、消息或服务发生了什么变化。
- 这一步为什么会推动下一步。

## 1. 用户路径

作用：按顺序说明用户看到了什么、做了什么、为什么会继续走下去。

每个小节都必须先写文案说明，再贴对应步骤截图。截图要紧跟小节文字，不要集中放到文末，也不要只依赖“用户完整路径时序图”的总览图。

推荐结构：

```markdown
### 1.x 阶段名称

文案说明：用户看到了什么、做了什么、页面如何引导、底层逻辑是什么。

**对应截图 / 对应 GIF**

![步骤截图](图片)

[如果分析动态交互，这里也要插入对应 GIF 或连续关键帧]

图注：这张图证明了哪个入口、文案、交互或进度设计。
```

小标题要按真实路径命名，不要硬套固定模板。可以参考这些阶段，但只写案例里真实出现的部分：
- 触达阶段：用户第一次在哪里看到入口。
- 进入阶段：用户进入了哪个页面、内容或场景。
- 决策阶段：用户被什么信息、利益点、信任点或限制条件推动。
- 行动阶段：用户完成了什么操作。
- 承接阶段：页面、消息、客服、社群、App、邮件、短信、直播间、门店或其他渠道如何继续接住用户。
- 留存阶段：用户为什么会回来或继续参与。
- 传播阶段：用户为什么会分享、邀请或带来别人。
- 转化阶段：用户最终完成了什么业务目标。

每一步都要写：
- 用户动作。
- 页面、消息、视频帧或截图证据。
- 关键文案或关键 UI 机制。
- 关键动态交互，比如弹窗、提示条、动画、按钮状态变化、进度变化。
- 这一步的业务目的。
- 证据不足时标 `[推断]`。

## 2. 规则设计

作用：把案例背后的规则拆出来，说明用户为什么会按这个路径行动。

规则不一定都是奖励或邀请，也可能是价格、门槛、时间、身份、库存、任务、内容分发、服务承诺、社交关系或平台限制。根据案例实际情况看：
- 进入规则：谁能参与，入口在哪里，有没有门槛。
- 行动规则：用户需要完成什么动作，动作顺序是否被设计过。
- 激励规则：有没有奖励、折扣、权益、内容、身份、排名、稀缺名额等。
- 约束规则：有没有时间限制、次数限制、资格限制、库存限制、地域限制等。
- 传播规则：有没有分享、邀请、拼团、助力、转发、口碑扩散等设计。
- 留存规则：有没有签到、任务、订阅、复访、提醒、连续参与等设计。
- 防滥用规则：有没有能看到的防作弊、风控、合规或审核设计。

要分清楚哪些是直接观察到的，哪些只是 `[推断]`。

## 3. 玩法解析

作用：解释用户为什么愿意参与、为什么愿意继续、为什么愿意完成目标动作。

每个小节都要先讲机制，再补一段“底层逻辑分析”。如果这一步有动态 UI，还要补一段“UI 交互分析”。最后贴对应截图、GIF 或连续关键帧。证据必须放在对应小节文字下方，不能只放在总览图、附件或文末。格式建议：

```markdown
玩法说明：这个玩法做了什么，用户在这一步看到了什么、被什么推动继续。

底层逻辑分析：这一步利用了哪些心理或行为机制，比如损失厌恶、收益放大、沉没成本、临门一脚、稀缺感、随机奖励、社交证明等。要结合截图里的文案、数字、按钮、进度或奖励来说明。

UI 交互分析：这一步的动态交互如何出现、突出什么、希望用户下一步做什么、业务目的是什么、有没有打扰或误导风险。证据不足时标 `[推断]`。

**证据截图**

![截图说明](图片)

图注：这张图证明了哪个入口、文案、交互、进度、奖励、风控或转化设计。
```

不要把截图集中堆在最后。读者看到某个玩法时，下面就应该能看到对应画面。`用户完整路径时序图` 是总览图，不能替代本节的小节截图。

重点看真实案例里出现的动机，不要预设一定是某一种玩法。可关注：
- 入口诱因：用户为什么会点进去。
- 利益驱动：用户获得什么好处，可能是省钱、赚钱、效率、身份、内容、情绪价值、社交价值、服务便利等。
- 信任机制：用户为什么相信它，可能来自品牌、达人、朋友、平台背书、评价、销量、真实案例、证书、门店、客服等。
- 降低阻力：流程是否短、说明是否清楚、是否减少下载/注册/填写/支付等成本。
- 紧迫感：是否有倒计时、限量、限时、库存、名额、阶段任务等。
- 社交机制：是否利用熟人关系、群体氛围、从众、攀比、协作或竞争。
- 内容机制：是否通过视频、图文、直播、教程、测评、评论、榜单等内容推动用户。
- 渠道闭环：用户是否被某个渠道持续承接，比如 App、网页、短信、邮件、IM、社群、直播间、门店、客服等。

底层逻辑分析要在上面这些拆解基础上增加，不要替换掉原来的分析。可以重点看：
- 损失厌恶：用户是否已经拿到、选中、积累或接近某个权益，所以不继续就像“亏了”。
- 收益放大或贪婪刺激：页面是否把奖励、积分、金额、概率、翻倍、大奖说得很突出，让用户觉得“再做一步可能赚更多”。
- 沉没成本：用户是否已经投入了时间、点击、选择、任务进度或社交关系，导致更不愿意放弃。
- 临门一脚：进度是否被设计成只差一点，比如 4/5、还差 1 人、还差 50 分，让用户觉得马上能完成。
- 随机奖励：奖励是否带有不确定性，比如抽奖、宝箱、翻牌、随机积分，让用户想继续试。
- 稀缺和紧迫：是否有倒计时、限时、库存、名额、阶段截止，推动用户马上行动。
- 社交证明和攀比：是否展示他人参与、好友助力、排行榜、成功案例，让用户觉得大家都在做。
- 互惠和人情压力：是否利用好友帮忙、助力回报、群体协作，让用户不好意思中断。

底层逻辑不能硬套名词。必须说明它对应到截图里的哪个文案、数字、按钮、进度、奖励或页面反馈；证据不足时标 `[推断]`。

UI 交互分析要拆清楚这几件事：
- 触发条件：用户做了什么，或系统在哪个状态下弹出。
- 出现位置：它盖住了哪里，靠近哪个关键金额、按钮、任务或进度。
- 视觉强调：颜色、动效、遮罩、红框、震动、音效、数字变化、按钮状态如何吸引注意。
- 用户下一步：它希望用户点什么、继续什么、放弃什么。
- 业务目的：它是在催促、解释、降低疑虑、制造紧迫感、强化收益，还是做风控提示。
- 打扰和风险：是否遮挡主路径、是否误导、是否让用户反感，证据不足时标 `[推断]`。

例如“今日提现峰值提醒”这种顶部提示，不能只写“系统提醒用户”。要写清楚它为什么在用户接近 50 元提现门槛时出现，如何借“当前处于提现高峰期、已有大量用户提现成功”制造紧迫感和社交证明，并把用户注意力拉回“今日成功提现”。

只要写了 UI 交互分析，就必须在该小节下方贴对应 GIF 或连续关键帧截图。不能只写“见用户完整路径时序图”，也不能只把 GIF 放在本地文件或附件里。

涉及数值设计时，要单独写清楚：
- 用户看到了哪些数字，比如奖励、进度、倒计时、门槛、次数、金额、百分比、排名、库存等。
- 这些数字是如何变化的，比如递增、递减、随机、分段、临门一脚、阶梯解锁等。
- 这些数字让用户产生了什么判断，比如“很快就能完成”“已经投入太多”“下一步更值得做”。
- 没有后台证据时，只能写前台可见数值和 `[推断]` 的产品意图。

## 4. 增长内核与归因

作用：把真正带来增长的原因和表面动作分开。

重点写：
- 核心增长引擎：到底是什么在驱动新增、留存、转化或传播。
- 增长循环：用户行为如何带来下一批用户、下一次访问、下一次购买或下一次内容消费。
- 关键杠杆：价格、渠道、内容、关系链、产品体验、供给、服务、品牌、平台流量等，哪个最关键。
- 归因拆分：哪些增长来自渠道，哪些来自产品机制，哪些来自运营动作，哪些来自外部环境。
- 证据等级：有数据的写事实，没有数据的标 `[推断]`。

不要把“看起来热闹”的动作直接当成增长内核，要说明它为什么真的能带来业务结果。

## 5. SaaS 后台逻辑

作用：把案例翻译成可复用的后台能力，方便后续做产品设计或系统设计。

这里的“SaaS 后台逻辑”不是说每个案例都必须是 SaaS，而是把前台表现翻译成后台需要支持的能力。如果目标产品不是 SaaS，也要写成可复用的系统能力。

根据案例实际情况拆：
- 用户与身份：注册、登录、授权、会员、角色、标签、分层等。
- 内容与页面：落地页、活动页、商品页、表单、弹窗、素材、评论、榜单等。
- 规则配置：价格、优惠、任务、门槛、权益、时间、库存、名额、状态流转等。
- 关系与传播：邀请关系、分享链路、拼团关系、助力关系、推荐来源、口碑来源等。
- 消息触达：站内信、短信、邮件、IM、社群、App 推送、客服、直播提醒等。
- 交易与转化：订单、支付、退款、订阅、续费、线索、预约、咨询、下载等。
- 数据与归因：漏斗、来源、转化率、留存、复购、分享、ROI、异常数据等。
- 风控与合规：防作弊、权限、审核、隐私、平台规则、财务风险等。

只有在目标技术栈明确时，才补技术映射，比如具体模块名、表结构或后台页面。不要为了显得完整强行写技术栈。

## 6. 如何复用

作用：把案例变成我们可以执行的产品方案、运营动作或系统能力。

如果用户指定目标行业或业务，例如 casino、电商、游戏、SaaS、金融、教育等，本节必须按该行业重写，不能只写通用复用点。要说明哪些机制能照搬，哪些必须改造，哪些因为合规或风控不能做。

重点写：
- 哪些机制可以直接复用。
- 哪些机制只能改造后复用。
- 哪些机制不能复用，原因是什么。
- 复用到我们的产品里应该做成什么功能、页面、规则、内容、后台配置或运营流程。
- 不同市场、不同渠道、不同用户层级下需要怎么改。
- 第一版应该做多小，先验证什么。

不要只写“值得学习”。要写清楚应该做成什么、谁来用、怎么判断有效。

## 7. 商业闭环

作用：说明这个案例如何从用户行为变成业务结果。

商业结果不一定只有付费，也可能是线索、订单、下载、留资、广告收入、订阅、复购、品牌曝光、渠道沉淀、内容消费或供应链效率。根据案例实际情况写：
- 用户完成的业务目标是什么。
- 这个目标如何变成收入、成本下降、线索沉淀、用户资产或长期价值。
- 有没有免费入口、低价入口、付费产品、套餐、佣金、广告、服务费、订阅、复购等结构。
- 已验证的数据是什么；没有数据就标 `[推断]`。
- 用户路径、留存、传播和商业结果之间如何互相影响。

## 8. 不足分析

作用：在复用前先看清风险，避免只看到成功的一面。

重点看：
- 证据不足：哪些关键数据没有看到。
- 平台依赖：是否高度依赖某个平台、算法、渠道或账号。
- 人或品牌依赖：是否依赖创始人、达人、主播、销售、品牌 IP、门店或强内容能力。
- 供给依赖：是否依赖特殊商品、价格、库存、服务能力、履约能力或资源位。
- 市场限制：地域、语言、文化、用户习惯、支付方式、物流、法规等是否会影响复用。
- 合规风险：隐私、广告、抽奖、返利、金融、医疗、教育、平台规则等风险。
- 产品限制：流程复杂、体验割裂、转化步骤过多、用户阻力高等。
- 数据限制：漏斗数据、归因数据、留存数据、收入数据是否缺失。
- 护城河强弱：别人是否容易复制，复制后效果是否会下降。

如果案例涉及奖励、返利、抽奖、助力、邀请、游戏化、虚拟币、下注、充值或提现，要补一段完整风控措施，至少覆盖：
- 身份与设备：账号、手机号、设备、IP、支付工具、收货信息、实名信息如何判重。
- 行为异常：短时间高频点击、批量注册、异常邀请、同设备多账号、同好友反复助力等如何识别。
- 奖励安全：奖励发放门槛、冷却时间、延迟到账、人工复核、黑名单、额度上限。
- 合规边界：抽奖、返利、广告承诺、未成年人、地区法规、隐私授权、用户数据展示。
- 运营处置：异常用户如何提示、冻结、驳回、申诉和恢复。

## 对我们 SaaS 产品的核心启发

作用：把拆解结论变成产品优先级。

这里要从“案例现象”落到“我们下一步应该做什么”。建议按优先级写：
- P0：最应该先做，能最快验证核心价值。
- P1：第二层能力，能提高效率、转化或复用范围。
- P2：后续可选能力，等 P0/P1 验证后再做。

每一项都要写：
- 功能名称。
- 对应案例里的哪个证据或机制。
- 为什么重要。
- 建议归属模块或负责人。
- 第一版最小可用功能应该包含什么。
- 验证指标是什么。

不要把案例里的具体表现原样照搬成需求，要先抽象成机制，再落成适合我们产品的能力。

## 飞书交付注意事项

当输出到已有飞书文档时，要保护已有图片、画板和表格：

- 不要默认使用 `overwrite`。
- 不要为了插图方便删除后半篇再重建。
- 不要把图片 token 写进 Markdown 后声称已插图；飞书 `docs +update` 会跳过缺少 URL 的图片。
- 新建文档可以按“文字 -> 图片 -> 文字”的顺序追加，保证截图贴在对应小节下面。
- 已有文档如果要把本地图片插到指定小节下方，优先用新版 `docs +media-insert --selection-with-ellipsis` 按唯一文本锚点插图；旧版 CLI 不支持时，再确认使用 docx block API 或新建迭代版文档。
- 不能用覆盖全文、删除重建、图片堆文末来代替精确插图。发现当前工具不支持精确插图时，先停下来说明限制，等用户确认下一步。
